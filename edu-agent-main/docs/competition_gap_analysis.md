# 中国软件杯赛题完成度与缺口分析

## 1. 赛题简介

本赛题围绕“基于大模型的个性化资源生成与学习多智能体系统开发”，要求参赛团队面向高等教育场景，构建一个能够理解学生学习需求、生成个性化学习资源、规划学习路径并持续评估学习效果的智能体系统。

赛题的核心目标是借助大模型、多智能体协作、多模态生成、知识库检索、内容安全过滤等技术，突破传统标准化教学难以适配学生差异的问题，为不同专业、不同基础、不同学习目标的大学生提供定制化学习支持。

系统应能够通过自然语言对话构建学生画像，结合课程知识库与学生薄弱点，生成课程讲解文档、思维导图、题库、拓展阅读、视频/动画、代码或实验案例等多类型学习资源，并根据学习过程和答题表现动态调整学习路径，实现面向高等教育的“因材施教”。

## 2. 赛题实用价值

该系统面向高校学生自主学习、课程复习、考试备考、实践训练等真实场景，具有以下价值：

- 降低学生从海量学习资料中筛选有效资源的成本。
- 依据学生专业、年级、基础、目标和薄弱点生成个性化资源。
- 通过多智能体分工提升资源生成的结构化程度和可解释性。
- 将课程讲解、知识图谱、练习题、实操案例、拓展阅读等资源统一组织成学习路径。
- 通过学习效果评估和动态画像更新，形成持续优化的学习闭环。
- 为高校课程数字化、智能化教学辅助系统提供可落地原型。

## 3. 涉及技术

当前项目已采用或预留的技术包括：

- Python 3.10+
- FastAPI
- Uvicorn
- Pydantic
- LangGraph
- Mock LLM
- RAG 关键词检索
- Chroma 向量库扩展接口
- Mermaid mindmap
- Markdown 教学内容生成
- SQL 实操案例生成

赛题完整版本还应进一步补齐：

- 真实大模型接入，优先考虑科大讯飞星火 API。
- 多模态资源生成，如 PPT、视频、动画、图解。
- 前端交互界面，支持 Markdown 渲染、流式输出和资源卡片展示。
- 用户画像持久化、学习行为记录和动态更新。
- 更严格的防幻觉、安全过滤和知识库依据追踪。
- 响应进度追踪、异步任务队列或流式返回机制。
- 完整测试、部署、系统开发说明书和测试说明书。

## 4. 当前项目已完成内容

当前 `edu-agent-ai` 已完成一个可运行的 backend-ai 基础框架，主要覆盖赛题的 AI 服务后端和多智能体主链路。

已完成能力：

- FastAPI 后端服务框架。
- `/` 和 `/health` 服务状态接口。
- `/api/analyze` 学生画像提取接口。
- `/api/generate` 多 Agent 资源生成接口。
- `/api/evaluate` 学习效果评估接口。
- `/api/courses` 课程列表接口。
- `/api/workflow` 工作流说明接口。
- `ProfileAgent` 学生画像 Agent。
- `RetrieverAgent` 知识库检索 Agent。
- `PlannerAgent` 学习路径规划 Agent。
- `DocumentAgent` Markdown 讲解文档 Agent。
- `MindMapAgent` Mermaid 思维导图 Agent。
- `QuizAgent` 多题型练习题 Agent。
- `PracticeAgent` SQL 实操案例 Agent。
- `SafetyCheckAgent` 安全检查 Agent。
- `EvaluatorAgent` 学习效果评估 Agent。
- LangGraph 工作流与 Python 顺序 fallback 工作流。
- 数据库系统课程初始知识库 10 篇 Markdown。
- Mock LLM 模式，无需真实 API Key。
- ChromaVectorStore 扩展占位类。
- README、接口示例、工作流说明和基础测试文件。

## 6. 目前最关键缺口

### 6.1 缺前端演示系统

赛题非功能需求明确提到界面美观、交互清晰、Markdown 渲染、流式输出、多模态内容卡片化展示。当前项目只有 AI 后端，没有前端，因此比赛演示完整度不足。

建议补充：

- 学生需求输入页。
- 学生画像展示卡片。
- 学习路径时间线。
- Markdown 文档渲染区。
- Mermaid 思维导图渲染区。
- 题库练习与答题提交区。
- SQL 实操案例展示区。
- 安全检查报告区。
- 生成进度条或流式输出效果。

### 6.2 缺真实大模型接入

当前使用 Mock LLM，适合开发阶段，但比赛最终需要体现大模型能力。赛题还提到开发过程中使用的 AI 辅助工具需选用科大讯飞相关工具，因此建议优先接入讯飞星火 API。

建议补充：

- 实现 `LLMClient.spark_generate`。
- 增加 prompt 模板管理。
- 增加结构化 JSON 输出解析与重试机制。
- 增加 LLM 调用日志、耗时统计和错误兜底。
- 保留 mock 模式用于测试和离线演示。

### 6.3 缺多模态资源

当前资源类型主要是文本、Mermaid 和 SQL。赛题强调多模态资源，如 PPT、视频、动画、图解等。

建议补充：

- PPT 大纲和可下载 PPT 文件生成。
- 知识点图解生成。
- 教学视频脚本生成。
- 动画分镜脚本生成。
- 若时间允许，接入多模态生成工具或先做“视频/动画脚本 + 静态图解”的可演示版本。

### 6.4 学生画像还不是动态画像

当前画像由单次输入规则抽取，尚未结合学习历史、练习情况和资源反馈持续更新。

建议补充：

- 用户画像存储表。
- 学习会话记录。
- 答题记录与错题记录。
- 资源浏览/收藏/完成状态。
- `/api/profile/{user_id}` 查询画像。
- `/api/profile/update` 根据行为更新画像。

### 6.5 RAG 知识库能力较弱

当前是关键词检索，知识库内容较短，尚不能充分支撑防幻觉和高质量讲解。

建议补充：

- 扩展数据库系统完整课程知识库。
- 将知识库拆分为章节、知识点、例题、实验、常见错误。
- 接入 Chroma 做向量检索。
- 返回引用来源和片段。
- 在生成内容中标注依据来源。

### 6.6 防幻觉与安全机制不足

当前 SafetyCheckAgent 只检查是否有检索资料和课程范围，无法判断生成内容是否事实错误。

建议补充：

- 基于知识库引用的事实一致性检查。
- 敏感内容与不适合教学内容过滤。
- SQL 安全检查，避免危险操作示例。
- 输出置信度和依据不足提示。
- 增加 EvaluatorAgent 对生成内容质量的二次评审。

### 6.7 缺异步任务与流式响应

资源生成可能耗时较长，赛题要求避免长时间白屏等待。当前接口是同步一次性返回。

建议补充：

- Server-Sent Events 流式输出。
- WebSocket 生成进度推送。
- 任务 ID + 状态查询接口。
- 后台任务队列，如 Celery、RQ 或 FastAPI BackgroundTasks。

### 6.8 缺标准比赛交付文档

当前 README 适合开发启动，但还不够覆盖赛题文档要求。

建议补充：

- 系统开发说明书。
- 需求分析说明书。
- 架构设计说明书。
- 数据库设计说明书。
- 接口设计说明书。
- 测试说明书。
- 部署说明书。
- 开源组件与协议说明。
- 演示脚本与答辩 PPT 提纲。

## 7. 建议后续开发优先级

### 第一阶段：形成可演示闭环

目标是让项目从“后端接口框架”变成“可以给评委完整演示的系统”。

优先任务：

1. 增加前端页面。
2. 增加 Markdown 和 Mermaid 渲染。
3. 增加学习路径、题库、SQL 案例的卡片化展示。
4. 增加 `/api/tutor` 智能辅导接口。
5. 增加生成进度或流式输出。

### 第二阶段：接入真实大模型和增强 RAG

目标是体现赛题中的大模型与知识库融合能力。

优先任务：

1. 接入科大讯飞星火 API。
2. 增加 prompt 模板。
3. 接入 Chroma。
4. 扩展数据库系统知识库。
5. 增加引用来源和防幻觉检查。

### 第三阶段：补齐个性化学习闭环

目标是从一次性资源生成升级为持续学习系统。

优先任务：

1. 增加用户、画像、学习记录、答题记录数据表。
2. 增加画像动态更新。
3. 增加错题分析和下一轮资源推荐。
4. 增加学习效果趋势分析。
5. 增加资源反馈机制。

### 第四阶段：补多模态和比赛文档

目标是提升创新性和交付完整度。

优先任务：

1. 生成 PPT 大纲或 PPT 文件。
2. 生成视频脚本、动画分镜和图解资源。
3. 整理系统开发说明书。
4. 整理测试说明书。
5. 整理开源组件与协议说明。
6. 准备演示脚本、答辩 PPT 和系统截图。

## 10. 最小可参赛版本建议

如果时间有限，建议至少补齐以下内容：

1. 一个可演示前端。
2. 星火 API 接入。
3. Chroma 向量检索。
4. 动态画像存储。
5. 智能辅导接口。
6. 生成进度展示。

10. 测试说明书。
